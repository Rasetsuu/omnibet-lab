# v971-v980 GUI model eval status

This milestone connects the v961 Rust baseline evaluation report to the normal Matches GUI.

## Purpose

The normal Matches data-pipeline card can now read both:

```text
reports/feature_counts.json
reports/model_eval.json
```

and display them together.

## Why this comes before bigger data

Before importing 1k-3k and 10k-30k rows, the GUI should show the complete readiness chain:

```text
row count
→ count gate readiness
→ baseline evaluation status
→ eval metrics
→ real model still locked until stronger gates pass
```

This prevents the app from becoming a silent report generator and keeps the normal GUI honest.

## GUI behavior

The Matches card keeps the existing safe fallback:

```text
Completed row count: 3 / 200 required for v1
V1 readiness: Needs more rows
Baseline eval: No baseline eval report loaded
Real model: Locked until enough settled rows
```

When `reports/feature_counts.json` is found, it updates row-count/readiness fields.

When `reports/model_eval.json` is found, it updates:

- baseline eval source
- baseline eval status
- eval metrics
- real-model lock/eval status

## Merge behavior

The GUI merges generated status reports instead of replacing one report with another.

This matters because feature counts and model eval reports are generated by different commands:

```text
omnibet-feature-count-gate
omnibet-baseline-eval
```

Both reports should be visible together when both exist.

## Metrics shown

When available, the GUI displays:

- accuracy
- log loss
- Brier score
- calibration ECE

The v961 report is still only a simple chronological expanding-frequency baseline.

## Product safety

The GUI still hides training/import controls from the normal match screen.

Even when baseline evaluation completes, the normal model status remains locked unless a future stronger evaluation gate explicitly sets:

```text
real_model_ready: true
```

No profit claims, staking output, or betting advice are introduced by this milestone.
